## 1. 规格访谈

**适合**：有想法但不确定怎么表达的任何人

**效果**：Claude 会追问关键细节，帮你整理出清晰的需求规格文档

```
你先不要写代码。

请阅读：CLAUDE.md、docs/spec_packet_template.md、skills/README.md

现在对我提供的需求进行 Spec Interview：
1. 提出 12~20 个关键追问（用户体验/验收、兼容、隐私、性能/成本、上线灰度与观测）
2. 等我回答后，整理为【Product Spec】（只写用户可见与验收，不写技术黑盒）
3. 生成【Tech Brief 草案】并标记守门员决策点（含风险等级）
4. 列出建议引用的 skills 文件清单

我的需求是：
[在这里描述你的需求]
```

## 2. 技术决策

**适合**：需要做技术选型、或不确定实现方案的任何人

**效果**：Claude 会分析选项、对比权衡、给出推荐方案

```
你先不要写代码。

请阅读：CLAUDE.md、docs/approved_components.md、docs/adr_template.md、skills/skill_tech_discovery_and_decision.md

1. 先提出关键追问（数据/隐私、质量定义、延迟、成本、可靠性、兼容性）

   - Problem
   - Constraints
   - Options（含 Default/Golden Path）
   - Trade-offs
   - Recommendation
   - Evaluation
   - Rollout
   - Open Questions
3. 检查是否偏离 approved_components.md 的默认方案，如偏离需写 ADR

我的问题是：
[在这里描述你的技术问题]
```

## 3. 全栈开发

**适合**：需要同时实现前端和后端的完整功能

**效果**：Claude 会协调前后端，先出整体计划再分别实现

```
请阅读：CLAUDE.md、skills/skill_api_contract.md、skills/skill_be_endpoint_go_gin.md、skills/skill_fe_guidelines.md

现在帮我实现一个完整功能（前端+后端）：

1. 先输出【整体 Implementation Plan】：
   - API 契约设计（请求/响应结构）
   - 后端改动（文件清单、影响面、风险等级）
   - 前端改动（组件结构、状态管理）
   - 依赖关系与实现顺序

2. 等我确认后，先实现后端：
   - 按 skill_be_endpoint_go_gin.md 规范
   - 输出 Test Plan

3. 后端完成后，实现前端：
   - 按 skill_fe_guidelines.md 规范
   - 处理各种状态（加载中/空/成功/失败/无权限）
   - 配合后端开关

4. 最后输出两个【MR Description】（前端和后端各一个）

我要实现的功能是：
[在这里描述完整需求]
```

4. 最后输出符合 MR 描述契约的【MR Description】

我要实现的功能是：

## 使用技巧

### 1. 提供足够的上下文

Claude 知道的越多，输出越准确：
- 相关的需求文档
- 涉及的代码文件
- 已有的设计决策
- 必须满足的约束

### 2. 分步确认

复杂任务时，让 Claude 先出计划，确认后再写代码：
- 避免方向错误导致返工
- 方便在 MR 前就发现问题

### 3. 及时纠正

如果 Claude 的理解有偏差，直接说：
- "不对，我的意思是..."
- "这里需要考虑..."
- "约束条件还有..."

### 4. 保持简洁

描述需求时：
- 说清楚"是什么"和"为什么"
- 不需要说"怎么做"（这是 Claude 的工作）

### 5. 善用守门员反馈

如果守门员指出问题：
1. 把反馈复制给 Claude
2. 让 Claude 理解问题并修改
3. 这样能学到最佳实践
