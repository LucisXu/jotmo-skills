# Jotmo Skills - AI 辅助开发知识库

## 这是什么

这是 Jotmo 团队的 **AI 辅助开发知识库**。通过这个知识库，团队中的**任何人**都可以借助 Claude 独立完成完整的需求交付。

**核心理念**：每个人都能独立走完"需求 → 实现 → 测试 → 上线"的全流程，代码守门员负责质量把关。

---

## 新协作模式

### 以前的方式
```
产品经理 → 设计师 → 前端开发 → 后端开发 → 测试 → 上线
   ↓         ↓         ↓          ↓
  需求      设计       页面       接口
```

### 现在的方式
```
任何人 + Claude → 完整实现 → 守门员审核 → 上线
                              ↓
                    前端守门员 / 后端守门员
```

**关键变化**：
- **不再有角色分工**：每个人都能独立完成需求，包括前端和后端
- **守门员制度**：曾经的前端/后端开发者，现在是代码质量守门员
  - 他们不再是"实现者"，而是"审核者"
  - 确保代码质量、架构一致性、最佳实践
  - 在 MR 阶段进行 Code Review

## 快速开始（3 步）

### 第 1 步：明确你要做什么

在和 Claude 对话前，先想清楚：
- 我要解决什么问题？
- 用户会怎么用这个功能？
- 有什么必须满足的约束？

### 第 2 步：选择合适的起手式

打开 `CLAUDE_PROMPTS.md`，找到适合你场景的提示词模板，复制后开始对话。

| 我要做什么 | 用哪个模板 |
|-----------|-----------|
| 把想法变成规格文档 | **规格访谈** |
| 技术选型或不确定怎么做 | **技术决策** |
| 实现一个完整功能（前端+后端） | **全栈开发** |
| 只改前端部分 | **前端实现** |
| 只改后端部分 | **后端实现** |
| 提交代码前检查 | **MR 准备** |

### 第 3 步：对话中确认再动手

Claude 会先输出计划，**等你确认后才写代码**。这样可以：
- 确保方向正确，避免返工
- 保证符合团队规范
- 在 MR 前就解决潜在问题

## 守门员职责

### 什么是守门员

守门员是代码质量的最后一道防线。他们负责：

| 职责 | 说明 |
|------|------|
| Code Review | 审核 MR 中的代码质量 |
| 架构把关 | 确保符合团队架构约定 |
| 最佳实践 | 指导和纠正不规范的实现 |
| 知识沉淀 | 发现问题后更新 skills 库 |

### 守门员不做什么

- ❌ 不再是唯一的实现者
- ❌ 不需要从头写每一行代码
- ❌ 不应该成为瓶颈

### 两类守门员

| 守门员 | 关注范围 |
|--------|---------|
| **前端守门员** | 页面/组件/状态管理/用户体验 |
| **后端守门员** | 接口/服务/数据库/性能/安全 |

## 常见场景速查

### 场景 1：我有个需求，不知道怎么开始

1. 打开 `CLAUDE_PROMPTS.md`，使用"规格访谈"模板
2. 把你的想法告诉 Claude
3. Claude 会追问细节，帮你整理出规格文档
4. 确认规格后，继续让 Claude 帮你实现

### 场景 2：我要完整实现一个功能

1. 使用"全栈开发"或具体的"前端/后端实现"模板
2. 告诉 Claude 你要做什么
3. Claude 会先出计划，你确认后再写代码
4. 代码完成后，使用"MR 准备"模板准备提交
5. 提交给对应的守门员审核

### 场景 3：我不确定技术方案怎么选

1. 使用"技术决策"模板
2. 描述你的问题和约束
3. Claude 会对比分析选项，给出推荐
4. 如果偏离默认方案，需要写 ADR 文档

### 场景 4：守门员发现我的代码有问题

1. 理解守门员的反馈
2. 把反馈告诉 Claude，让它帮你修改
3. 修改后重新提交 MR

## 团队底线（必须遵守）

无论谁做开发，都要确保：

1. **线上稳定** - 不引入已知问题
2. **向后兼容** - 老用户、老数据不受影响
3. **可灰度** - 能小范围先试
4. **可回滚** - 出问题能快速恢复
5. **可观测** - 知道系统在发生什么

这些规则写在 `CLAUDE.md` 中，Claude 会自动遵守。

## 遇到问题怎么办

### 问：Claude 写的代码不对怎么办？

告诉 Claude 哪里不对，它会修正。如果反复不对，检查：
- 需求描述是否清晰
- 是否遗漏了重要约束

### 问：守门员不同意我的实现方式

1. 理解守门员的顾虑
2. 和守门员讨论，达成共识
3. 让 Claude 按新方案重新实现

### 问：我的场景没有对应的模板

使用"规格访谈"模板，描述你的需求，Claude 会帮你找到合适的方式。

## 持续改进

每次遇到问题，问三个问题：

1. **是规则缺失？** → 补充到 `CLAUDE.md`
2. **是套路缺失？** → 补充到 `skills/`
3. **是文档过时？** → 更新 `docs/`

守门员发现的共性问题，应该沉淀到 skills 库，让所有人受益。
